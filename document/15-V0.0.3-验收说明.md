### 1.1 需求条目（V0.0.3）

| ID | 描述 | 验收要点 |
|----|------|----------|
| P1 | 文件推送至网关已知路径 | 简报 `FullPathFileName` 即落盘路径；须在 `allowedPathRoots` 内 |
| P2 | 先推送简报，确认成功后再推送内容 | 简报 `Status=1` 时平台不得再发内容；网关不建会话 |
| P3 | 每帧内容后网关发内容确认；超时/断连则失败 | 段序号连续；`timeoutSec` 内无后续帧则关句柄；MQTT 断开等同失败 |
| P4 | 不考虑断点续传 | 无 `StartByte`；中断后须新 `CmdId` 全量重推 |

## 3. 目标机手工验收（推送）

### 3.1 环境与配置

| 检查项 | 说明 |
|--------|------|
| Broker | 开发机 mosquitto，目标机 `brokerHost` 指向开发机 IP |
| `gatewayId` | `transferFile.platform.json` 与 `transferFile.gateway.target.json` 一致（默认 `gw001`） |
| `allowedPathRoots` | 目标机配置须包含推送保存路径（如 `/tmp/`、`/data/transfer/`） |
| 二进制 | `build-openwrt/transferFile` 为 V0.0.3 重新交叉编译 |

### 3.2 全量推送（P1～P3）

**目标机**：

```bash
# 确保目录在 allowedPathRoots 内
./transferFile -c transferFile.gateway.target.json
```

**开发机**：

```bash
dd if=/dev/urandom of=/tmp/v003_push_src.bin bs=10240 count=1
./build/platform_sim -c config/transferFile.platform.json \
  --push-file /tmp/v003_push_src.bin \
  --gateway-path /tmp/v003_push_dst.bin
```

**预期**：

1. 平台日志：`网关接受推送` → 多段 `已发布推送内容` → `推送完成`
2. 网关日志：`推送简报确认成功` → 各段 `推送内容确认` → `推送文件接收完成`
3. 目标机校验：

```bash
md5sum /tmp/v003_push_src.bin   # 开发机
md5sum /tmp/v003_push_dst.bin   # 目标机，应与源一致
```

### 3.3 P2：简报失败不发内容

开发机推送至目标机**不允许**的路径（如 `/etc/passwd`，或不在 `allowedPathRoots`）：

```bash
./build/platform_sim -c config/transferFile.platform.json \
  --push-file /tmp/v003_push_src.bin \
  --gateway-path /etc/v003_should_fail.bin
```

**预期**：平台 `网关推送简报确认失败`；目标机无 `/etc/v003_should_fail.bin`；网关日志含 `INVALID_PATH` 或类似失败码。

### 3.4 P3：超时（spot check）

1. 临时将目标机配置 `timeoutSec` 改为 `10`（或开发机推送大文件后**人为暂停**不发下一段）。
2. 推送开始后中断平台侧发送，等待超过 `timeoutSec`。
3. **预期**：网关日志 `推送接收中止` / 超时；目标机侧**不完整**文件可删除；再次推送须使用**新 CmdId** 全量重推（P4）。

### 3.5 与召唤互斥（BUSY）

1. 目标机先进行**召唤上传**（`--gateway-file`，传输进行中）。
2. 开发机同时发起 `--push-file`。
3. **预期**：推送简报确认 `Status=1`，`ErrorCode=BUSY`（或召唤侧 BUSY，视先后顺序而定）。
