# 16 — 技术总结（算法、编码、功能与难点）

**文档版本**：V0.0.4（2026-06-03）  
**适用对象**：架构评审、交接、对外技术说明

---

## 1. 项目概述

| 项 | 说明 |
|----|------|
| 名称 | transferFile（网关侧 MQTT 文件传输） |
| 当前版本 | **V0.0.4** |
| 交付形态 | 目标机可执行 `transferFile` + 开发机联调工具 `platform_sim` |
| 传输通道 | MQTT（libmosquitto / 内存模拟总线） |
| 载荷格式 | UTF-8 JSON，`Data` 下字段均为 **string** |
| 开发方法 | TDD（47 项自动化测试） |
| 目标平台 | 本机 x86_64 + OpenWrt aarch64 交叉编译 |

本项目实现**网关**在 MQTT 上的两类文件业务：

1. **召唤上传**（V0.0.4）：平台召唤 → 网关读本地文件 → 简报 + **逐段上传 + 平台内容确认**；支持 **断点续传**。
2. **平台推送**（V0.0.3）：平台推简报 + 分段内容 → 网关写入已知路径；**每段确认**；**无断点续传**。

## 2. 功能点清单

### 2.1 召唤上传（R1～R5）

| ID | 功能 | 实现模块 |
|----|------|----------|
| R1 | 订阅召唤 Topic，解析 JSON，创建/恢复会话 | `MosquittoMqttAdapter`、`JsonProtocolCodec`、`TransferOrchestrator` |
| R2 | 严格「先简报、后内容」；未简报成功不得 `publishContent` | `TransferOrchestrator` |
| R3 | 路径非法、文件不存在、BUSY 等仅简报失败，内容帧数 0 | 各失败分支 + 测试断言 |
| R4 | `timeoutSec`（默认 180s）超时清除会话、停止发送 | `TimeoutWatchdog` |
| R5 | `StartByte`（1-based）断点续传；`FileSegNo` 每次新召唤从 1 重计 | `TransferOrchestrator`、`06-断点续传设计` |

### 2.1.1 召唤内容确认（C1～C3，V0.0.4）

| ID | 功能 | 实现模块 |
|----|------|----------|
| C1 | 订阅 `platform/content_confirm`，解析确认 JSON | `MosquittoMqttAdapter`、`decodeContentConfirm` |
| C2 | `WaitingContentConfirm` 态；成功确认后 `sendNextSegment` | `TransferOrchestrator::onContentConfirm` |
| C3 | 确认失败或超时 `abortSession` | Orchestrator + Watchdog |

### 2.2 平台推送（P1～P4）

| ID | 功能 | 实现模块 |
|----|------|----------|
| P1 | 按简报 `FullPathFileName` 创建/覆盖写文件 | `FileStore::openWriteCreate`、`writeAt` |
| P2 | 简报确认失败则不再收内容 | `PushReceiveOrchestrator::onPushBrief` |
| P3 | 每段 `FileSegNo` 顺序校验，逐段内容确认；超时关写句柄 | `PushReceiveOrchestrator`、`TimeoutWatchdog` |
| P4 | 不支持续传；段错/超时须新 CmdId 全量重推 | 会话状态机设计 |

### 2.3 工程与运维能力

| 能力 | 说明 |
|------|------|
| JSON 配置 | Broker、Topic、`timeoutSec`、`chunkSize`、`allowedPathRoots` |
| 双模式 MQTT | `useSimulatedBus` 内存总线 / 真实 mosquitto |
| 运行日志 | 控制台 + `log/YYYY-MM-DD.log`，关键步骤附带 JSON 报文 |
| 编译时间戳 | 每次构建刷新 `kBuildDateTime`（`transfer_refresh_build_info`） |
| 交叉编译打包 | `build-openwrt.sh`、`package-target.sh` |
| 平台联调 | `platform_sim`：`--gateway-file`、`--start-byte`、`--push-file` |

### 2.4 会话互斥

召唤与推送**不能同时进行**：双方 `setBusyChecker`，忙时返回 `BUSY`，避免单进程多传输竞态。

## 3. 编码与数据表示

### 3.1 协议层编码

| 编码/格式 | 用途 | 实现位置 |
|-----------|------|----------|
| **UTF-8 JSON** | 全部 MQTT 载荷 | `json_codec.cpp`、`json_push_codec.cpp` |
| **字符串数值** | `CmdId`、`FileSize`、`StartByte` 等协议均为 string，网关解析为 `uint32`/`uint64` | 手写 `extractJsonStringField` + `parseU32`/`parseU64` |
| **Base64** | 内容段 `Content` 字段承载二进制 | `base64.cpp` |
| **十六进制 CRC** | 简报 `FileCrc`，如 `0xCBF43926` | `Crc32Calculator::toHexString`、`parseFileCrcHex` |
| **时间字符串** | `ModifyTime`：`YYYY-MM-DD HH:MM:SS` | `FileStore::getModifyTime` |

### 3.2 Base64 说明

- **召唤/推送内容段**：原始文件字节 → Base64 → 填入 JSON `Content`。
- 示例：`"Content":"dGVzdAo="` 解码为 `test\n`（5 字节）。
- **注意**：大文件单段 Base64 很长；日志中对报文做 **2048 字节截断**，避免刷屏。

### 3.3 字节序与偏移约定

| 概念 | 约定 |
|------|------|
| `StartByte` | **1-based**（协议）；网关内部 `fileOffset = StartByte - 1` |
| `FileSegNo` | **1-based**，每段递增；续传时新召唤序列从 1 重新开始 |
| 文件读写 | `readAt` / `writeAt` 使用 **0-based** 偏移 + `lseek` |

### 3.4 错误码（string）

占位常量见 `error_codes.hpp`，如 `BAD_FRAME`、`INVALID_PATH`、`BUSY`、`CRC_MISMATCH`、`INVALID_SEG_NO`、`NO_SESSION` 等，编码进简报/确认的 `ErrorCode` 字段。

## 4. 算法与核心逻辑

### 4.1 CRC32（文件完整性）

| 项 | 内容 |
|----|------|
| 算法 | CRC-32 **reflected**，多项式 **0xEDB88320** |
| 初值/异或 | 初值 `0xFFFFFFFF`，结果 `^ 0xFFFFFFFF` |
| 用途 | 召唤简报成功时上报 `FileCrc`；推送收齐末段后校验与简报一致 |
| 实现 | 按字节位反射更新（`crc32Update`）；读文件 64KB 分块流式计算 |
| 输出格式 | `0x` + 8 位大写十六进制 |

### 4.2 Base64 编解码

| 项 | 内容 |
|----|------|
| 标准 | RFC 4648 风格（实现于 `base64.cpp`） |
| 难点修复 | V0.0.2 修复**大文件分段解码多 1 字节**问题（尾块 padding 处理），保障 R5/大文件验收 |
| 测试 | `test_base64.cpp` 向量 + 大缓冲边界 |

### 4.3 路径规范化与安全校验

| 步骤 | 算法/规则 |
|------|-----------|
| 规范化 | 折叠重复 `/`、处理部分 `.` 段、去尾斜杠 |
| 越权检测 | 拒绝含 `..`；路径必须落在 `allowedPathRoots` 前缀下 |
| 读写 | POSIX `open`/`read`/`write` + `lseek`；推送创建用 `O_CREAT\|O_TRUNC` |

### 4.4 分段传输

**召唤上传（网关 → 平台）**

```
readLen = min(chunkSize, fileSize - currentOffset)
Continue = (currentOffset + readLen < fileSize) ? "1" : "0"
FileSegNo 从 1 递增
```

**平台推送（平台 → 网关）**

```
按平台下发的 FileSegNo 严格顺序写入
nextSegNo 会话内递增；不匹配则 INVALID_SEG_NO 并 abort
Continue=0 时校验 bytesWritten == expectedFileSize 且 CRC 一致
```

### 4.5 超时监视（Watchdog）

| 项 | 内容 |
|----|------|
| 结构 | `cmdId → deadline` 映射，互斥锁保护 |
| 触发 | 主循环周期性 `tick()`；到期回调 `onTimeout(cmdId)` |
| 召唤 | 新合法召唤 `reset`；完成/失败 `disarm` |
| 推送 | 简报成功后 `arm`；每段确认后 `reset`；完成/中止 `disarm` |
| 测试 | 注入 `SteadyClock` Mock，秒级加速 R4 验收 |

### 4.6 JSON 解析策略

未引入第三方 JSON 库（减小嵌入式依赖与体积）：

- 针对固定字段名的 **轻量字符串扫描**（`extractJsonStringField`）。
- 优点：依赖少、可预测内存；缺点：不支持任意 schema 扩展，字段顺序无关但嵌套需符合约定。

## 5. 关键技术架构

### 5.1 分层与依赖倒置

```text
main
  → IGatewayMqtt（适配层：mosquitto / 模拟总线）
  → TransferOrchestrator / PushReceiveOrchestrator（应用层）
  → IProtocolCodec、IFileStore、ISessionStore、ITimeoutWatchdog（领域接口）
```

- 应用层**不链接**具体 MQTT 实现细节，便于单测 Mock。
- 领域层编解码、CRC、文件 IO **可独立 ctest**。

### 5.2 双编排器 + 共享基础设施

| 编排器 | 方向 | 会话存储 |
|--------|------|----------|
| `TransferOrchestrator` | 网关读文件 → 上传 | `MemorySessionStore` |
| `PushReceiveOrchestrator` | 平台下推 → 网关写文件 | `MemoryPushSessionStore` |

共享：`FileStore`、`TimeoutWatchdog`、`RuntimeLog`、MQTT 适配器。

### 5.3 MQTT Topic 模型

默认前缀 `transfer/sim/{gatewayId}/`：

| 业务 | 平台 → 网关 | 网关 → 平台 |
|------|-------------|-------------|
| 召唤上传 | `platform/summon`、`platform/content_confirm`（V0.0.4） | `gateway/brief`、`gateway/content` |
| 平台推送 | `platform/push/brief`、`platform/push/content` | `gateway/push/brief_confirm`、`gateway/push/content_confirm` |

QoS 默认 1；`gatewayId` 与 Topic 可通过 JSON 配置覆盖。

### 5.4 构建与部署技术

| 技术 | 说明 |
|------|------|
| CMake 3.14+ | `transfer_core` 静态库 + 可执行文件 |
| C++17 | GCC 8.3（OpenWrt 工具链） |
| `transfer_refresh_build_info` | ALL 目标，保证每次构建更新编译时间 |
| 交叉编译 | `BUILD_TESTING=OFF`，仅产物 `transferFile` + `libmosquitto.so*` |
| 部署包 | `dist/transferFile-target-aarch64.tar.gz` |

### 5.5 测试技术

| 层次 | 手段 |
|------|------|
| 单元测试 | Codec、FileStore、Orchestrator、Watchdog、Push、BuildInfo |
| 集成测试 | `SimulatedMqttBus` 端到端（无 Broker） |
| 验收 | `test_orchestrator_v002`（R3/R4/R5/大文件）、`test_orchestrator_v004`（C1～C3）、`test_push_*`（P1～P4） |
| 框架 | 系统 GTest 或自研 `minimal_test.hpp`（无 GTest 时） |

## 6. 技术难点与应对

| 难点 | 原因 | 本项目应对 |
|------|------|------------|
| **召唤逐段确认与超时** | V0.0.4 每段须等平台确认，否则不能发下一段 | `WaitingContentConfirm` + 确认成功/失败均 `reset`；推送同理逐段确认 |
| **1-based / 0-based 混用** | 协议 `StartByte`、`FileSegNo` 为 1-based，系统 IO 为 0-based | DTO 层显式 `fileOffset`；代码与文档双写换算公式 |
| **断点续传段号重置** | 续传是新召唤，平台按段序拼接而非带偏移字段 | 每次从 `FileSegNo=1` 发送；测试断言偏移与内容一致 |
| **Base64 尾块边界** | 分段长度非 3 倍数时 padding 易错 | 专项修复 + `test_base64` / 大文件用例 |
| **路径穿越与越权** | 网关可读/写本地任意路径风险 | `allowedPathRoots` 前缀白名单 + `..` 拒绝 |
| **单进程会话并发** | 嵌入式资源有限 | BUSY 互斥；召唤与推送 BusyChecker 交叉检查 |
| **推送顺序与部分写** | 网络乱序、重复段会破坏文件 | 严格 `FileSegNo == nextSegNo`；失败 `abortSession` 关 fd |
| **嵌入式依赖体积** | OpenWrt 存储紧张 | 静态链接核心逻辑；mosquitto 动态库随包部署；无重量级 JSON 库 |
| **双机联调复杂度** | Broker 在开发机、网关在板子 | 分平台/目标机配置、`platform_sim`、文档 10/15 |
| **编译时间显示陈旧** | CMake OUTPUT 规则不触发重生成 | 改为 `ALL` 自定义目标 `transfer_refresh_build_info` |

## 7. 创新点与工程亮点

> 说明：以下为**在本项目场景下的工程化创新/亮点**，非学术算法发明。

| 亮点 | 描述 |
|------|------|
| **双业务单进程统一网关** | 同一 `transferFile` 进程同时支持「上传」与「下推」两套 Topic/状态机，互不干扰且互斥保护。 |
| **接口化 + 双总线一键切换** | `useSimulatedBus` 在开发机无 Broker 即可跑通 47 项测试；目标机同一套代码走 mosquitto。 |
| **轻量 JSON 协议栈** | 零第三方 JSON 依赖，适合资源受限网关；协议字段与代码一一可追溯。 |
| **TDD 驱动的可验收版本** | V0.0.2/V0.0.3/V0.0.4 均有独立验收文档与命名测试（R3/R4/R5、P1～P4、C1～C3），回归可脚本化。 |
| **可注入时钟的超时测试** | 不依赖真实睡眠 180s，CI 秒级验证 R4。 |
| **联调可观测性** | 运行日志附带关键 JSON 报文（可截断），双机问题可对照协议逐帧排查。 |
| **双方向逐段确认** | V0.0.4 起召唤上传亦采用平台内容确认；推送由网关确认，降低半包/丢包导致静默损坏的风险。 |
| **OpenWrt 交付闭环** | 交叉编译、动态库打包、目标机配置模板与验收文档一体化。 |
