# 04 — 通信协议

## 1. 文档目的

定义网关与平台之间基于 MQTT 的消息通道及三类业务报文格式。载荷为 **JSON**，业务字段均位于根对象下的 **`Data`** 对象内；**JSON 中各字段类型均为 string**（数值亦以字符串传递）。

**协议版本**：`protocol-1.0`（召唤上传，2026-06-01）；`protocol-1.0 + push`（平台推送，2026-06-03，见 [14-V0.0.3-平台推送.md](14-V0.0.3-平台推送.md)）

## 2. MQTT 传输层

### 2.1 连接参数（`mqtt_config.hpp` + JSON 配置）

| 参数 | 说明 | 默认/实现 |
|------|------|-----------|
| Broker 地址 | host + port | 配置 `brokerHost` / `brokerPort`，默认 `127.0.0.1:1883` |
| ClientId | 客户端标识 | 空则 `gateway-transfer-sim-{gatewayId}` |
| gatewayId | 网关 ID | `gw001` |
| 用户名/密码 / TLS | 鉴权 | 字段预留，V0.0.1 未启用 |
| 传输模式 | `useSimulatedBus` | `true`：`SimulatedMqttBus`；`false`：`MosquittoMqttAdapter` |

### 2.2 Topic 约定（可配置，默认规则）

默认 `gatewayId=gw001` 时：

| 方向 | 用途 | 占位 Topic | QoS |
|------|------|------------|-----|
| 平台 → 网关 | 召唤文件 | `transfer/sim/gw001/platform/summon` | 1 |
| 网关 → 平台 | 文件简报 | `transfer/sim/gw001/gateway/brief` | 1 |
| 网关 → 平台 | 文件内容 | `transfer/sim/gw001/gateway/content` | 1 |
| 平台 → 网关 | **推送文件简报**（V0.0.3） | `transfer/sim/gw001/platform/push/brief` | 1 |
| 网关 → 平台 | **推送简报确认** | `transfer/sim/gw001/gateway/push/brief_confirm` | 1 |
| 平台 → 网关 | **推送文件内容** | `transfer/sim/gw001/platform/push/content` | 1 |
| 网关 → 平台 | **推送内容确认** | `transfer/sim/gw001/gateway/push/content_confirm` | 1 |

生成规则：`transfer/sim/{gatewayId}/platform/summon` 等（`makeDefaultMqttConfig()`）。推送流程详见 [14-V0.0.3-平台推送.md](14-V0.0.3-平台推送.md)。

### 2.3 载荷通用约定

| 项 | 约定 |
|----|------|
| 编码 | UTF-8 JSON |
| 结构 | 业务字段放在 `Data` 对象下；MQTT payload 为完整 JSON 文本 |
| 字段类型 | 协议层全部为 **string**；网关解析后转为内部数值类型 |
| 数值范围 | `CmdId`：0～4294967295（无符号 32 位，字符串十进制表示） |
| 字节位置 | **`StartByte` 从 1 开始计数**（1-based）；网关内部读写文件时转换为 0-based：`fileOffset = StartByte - 1` |
| 文件校验 | 简报成功时提供 **CRC32**（见 4.4 节算法） |
| 内容编码 | 文件分段内容在 JSON 中以 **Base64** 字符串置于 `Content` 字段 |

## 3. 召唤文件（平台 → 网关）

业务名称：**召唤文件**。

### 3.1 JSON 结构

```json
{
  "Data": {
    "CmdId": "12345",
    "FullPathFileName": "/aaa/bbb/ccc.ddd",
    "StartByte": "1"
  }
}
```

### 3.2 字段定义

| 字段路径 | 类型 | 必填 | 说明 |
|----------|------|------|------|
| Data/CmdId | string | 是 | 命令 ID，标识每一次文件请求/传输命令；取值范围 0～4294967295 |
| Data/FullPathFileName | string | 是 | 文件全路径名（或经平台/网关约定映射后的路径） |
| Data/StartByte | string | 是 | 本次传输在文件中的起始字节位置，**从 1 开始**；首次完整传输一般为 `"1"`；断点续传为待传首字节的 1-based 位置 |

### 3.3 网关处理要求

| 规则 | 说明 |
|------|------|
| JSON 解析失败 | 不发送内容；回复简报失败，`Status="1"`，`ErrorCode` 见第 7 节 |
| CmdId 校验 | 必须为合法无符号 32 位十进制字符串 |
| 路径安全 | 对 `FullPathFileName` 做规范化与越权检查（禁止 `..` 逃出允许根目录等，见 [02-系统架构.md](02-系统架构.md)） |
| StartByte | 解析为整数 \(s \ge 1\)；内部偏移 \(fileOffset = s - 1\)；须满足 \(fileOffset \le fileSize\) |
| 首次传输 | `StartByte="1"` 表示从文件第一个字节开始 |
| 断点续传 | `StartByte > 1` 时从对应偏移继续，见 [06-断点续传设计.md](06-断点续传设计.md) |
| 忙 | 已有其他活跃传输且非同一 CmdId 续传时，简报失败（如 BUSY） |

### 3.4 示例

```json
{
  "Data": {
    "CmdId": "1001",
    "FullPathFileName": "/data/logs/device001.log",
    "StartByte": "1"
  }
}
```

断点续传示例（从第 4097 字节开始传，1-based）：

```json
{
  "Data": {
    "CmdId": "1001",
    "FullPathFileName": "/data/logs/device001.log",
    "StartByte": "4097"
  }
}
```

## 4. 应答文件简报（网关 → 平台）

业务名称：**应答文件简报**。

### 4.1 JSON 结构

```json
{
  "Data": {
    "CmdId": "12345",
    "Status": "0",
    "ErrorCode": "",
    "Note": "",
    "FileCrc": "0x12345678",
    "FileSize": "1048576",
    "ModifyTime": "2026-06-01 12:00:00"
  }
}
```

### 4.2 字段定义

| 字段路径 | 类型 | 必填 | 说明 |
|----------|------|------|------|
| Data/CmdId | string | 是 | 与对应召唤命令的 `CmdId` 一致 |
| Data/Status | string | 是 | 执行状态：`"0"` 成功，`"1"` 失败 |
| Data/ErrorCode | string | 条件 | 失败时必填；成功时可空字符串 |
| Data/Note | string | 否 | 备注，可填写成功或失败说明 |
| Data/FileCrc | string | 条件 | **成功时必填**；文件 CRC32，十六进制字符串，如 `"0x12345678"` |
| Data/FileSize | string | 条件 | **成功时必填**；文件总大小，单位：字节 |
| Data/ModifyTime | string | 条件 | **成功时必填**；文件最后修改时间，格式 `YYYY-MM-DD HH:MM:SS` |

### 4.3 语义约束（R2、R3）

| Status | 网关后续行为 |
|--------|----------------|
| `"0"` | 允许进入文件内容发送阶段 |
| `"1"` | **不得**发送任何文件内容报文 |

### 4.4 文件 CRC32 算法

简报中的 `FileCrc` 对**整个文件内容**计算，规则如下：

| 项 | 说明 |
|----|------|
| 算法 | CRC32 |
| 生成多项式 | \(x^{32} + x^{26} + x^{23} + x^{22} + x^{16} + x^{12} + x^{11} + x^{10} + x^8 + x^7 + x^5 + x^4 + x^2 + x + 1\) |
| 校验方式 | **反向（Reflected）**校验，使用常量 **`0xEDB88320`**（即 IEEE 802.3 / PNG 常用的 reflected CRC-32） |
| 输出格式 | 十六进制字符串，建议带 `0x` 前缀，如 `"0xEDB88320"`（与平台展示格式保持一致） |

实现时建议以标准测试向量（如 `"123456789"` → `0xCBF43926`）验证与平台一致。

### 4.5 失败示例

```json
{
  "Data": {
    "CmdId": "1001",
    "Status": "1",
    "ErrorCode": "FILE_NOT_FOUND",
    "Note": "文件不存在",
    "FileCrc": "",
    "FileSize": "",
    "ModifyTime": ""
  }
}
```

## 5. 文件内容传输（网关 → 平台）

业务名称：**文件内容传输**。

### 5.1 JSON 结构

```json
{
  "Data": {
    "CmdId": "12345",
    "FileSegNo": "1",
    "Content": "SGVsbG8=",
    "Continue": "1"
  }
}
```

### 5.2 字段定义

| 字段路径 | 类型 | 必填 | 说明 |
|----------|------|------|------|
| Data/CmdId | string | 是 | 与召唤命令的 `CmdId` 相同 |
| Data/FileSegNo | string | 是 | 文件段序号，**从 1 递增**，直至本命令文件传输完毕 |
| Data/Content | string | 是 | 本段文件内容的 **Base64** 编码 |
| Data/Continue | string | 是 | 是否还有后续段：`"1"` 有续传段，`"0"` 无续传段（当前为最后一段） |

### 5.3 发送规则

| 规则 | 说明 |
|------|------|
| 顺序 | 仅在简报 `Status="0"` 成功后发送；**先简报，后内容**（R2） |
| 段序号 | 单次召唤触发的传输内，`FileSegNo` 从 `"1"` 递增，连续无跳号 |
| 数据范围 | 从召唤 `StartByte` 对应位置起，按配置的 `chunkSize` 分段读取文件并 Base64 编码填入 `Content` |
| 最后一段 | 最后一段置 `Continue="0"`，其余段为 `"1"` |
| 段与偏移 | 网关内部维护 `nextFileOffset`（0-based）；第 n 段覆盖的字节区间由 `StartByte` 与段大小决定，**内容报文不单独带文件偏移字段**，平台按段序拼接 |
| 失败 | 简报失败后内容报文数为 0（R3） |

### 5.4 分段逻辑（实现参考）

设召唤解析得到 `startOffset = StartByte - 1`（0-based），文件大小 `fileSize`，段大小 `chunkSize`：

1. `currentOffset = startOffset`，`segNo = 1`
2. 循环：`readLen = min(chunkSize, fileSize - currentOffset)`，若 `readLen == 0` 结束
3. 编码 `Content`，`Continue = (currentOffset + readLen < fileSize) ? "1" : "0"`
4. `currentOffset += readLen`，`segNo++`

### 5.5 示例（两段文件）

第一段：

```json
{
  "Data": {
    "CmdId": "1001",
    "FileSegNo": "1",
    "Content": "AQIDBA==",
    "Continue": "1"
  }
}
```

最后一段：

```json
{
  "Data": {
    "CmdId": "1001",
    "FileSegNo": "2",
    "Content": "BQYHCA==",
    "Continue": "0"
  }
}
```

## 6. 平台 ACK 帧

**当前协议未定义**平台对内容段的 ACK 报文。超时与续传见第 8 节。

## 7. 错误码（占位，代码见 `error_codes.hpp`）

`ErrorCode` 为 string。成功简报时可为空；失败时必填。**平台确认后统一替换常量定义。**

| ErrorCode | 说明 | 典型场景 |
|-----------|------|----------|
| `OK` | 成功（仅内部） | — |
| `BAD_FRAME` | JSON 无效或缺少必填字段 | 解析失败 |
| `INVALID_CMD_ID` | CmdId 非法 | 非数字或超范围 |
| `INVALID_START_BYTE` | StartByte 非法 | 超出文件大小 |
| `FILE_NOT_FOUND` | 文件不存在 | 打开失败 |
| `PERMISSION_DENIED` | 无读权限 | |
| `INVALID_PATH` | 路径非法或越权 | |
| `BUSY` | 网关忙 | 另有传输进行中 |
| `IO_ERROR` | 读盘错误 | |
| `FILE_CHANGED` | 文件已变更 | 续传校验（可选） |
| `CRC_MISMATCH` | CRC 不一致 | 推送收齐后校验失败 |
| `INVALID_SEG_NO` | 段序号错误 | 推送内容 `FileSegNo` 非预期 |
| `NO_SESSION` | 无会话 | 未简报成功即收内容等 |

## 8. 有效交互定义（与 R4 相关）

| 事件 | 是否重置超时计时器 |
|------|-------------------|
| 收到新的合法召唤（含续传） | **是** |
| 收到平台 ACK | **不适用**（协议无 ACK） |
| 网关发送内容段 | **否** |

**结论**：传输过程中若平台长时间无新召唤，依赖 **180s 超时** 中止；平台通过**再次召唤**并增大 `StartByte` 完成断点续传。

## 9. 报文对照（召唤上传 + 平台推送）

| 报文 | 方向 | 关键字段 |
|------|------|----------|
| 召唤文件 | 平台 → 网关 | CmdId, FullPathFileName, StartByte |
| 应答文件简报 | 网关 → 平台 | CmdId, Status, ErrorCode, FileCrc, FileSize, ModifyTime |
| 文件内容传输 | 网关 → 平台 | CmdId, FileSegNo, Content(Base64), Continue |
| 推送文件简报 | 平台 → 网关 | CmdId, FullPathFileName, FileCrc, FileSize, ModifyTime |
| 推送简报确认 | 网关 → 平台 | CmdId, Status, ErrorCode, Note |
| 推送文件内容 | 平台 → 网关 | CmdId, FileSegNo, Content(Base64), Continue |
| 推送内容确认 | 网关 → 平台 | CmdId, FileSegNo, Status, ErrorCode, Note |
